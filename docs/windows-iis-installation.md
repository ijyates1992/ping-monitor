# Windows + IIS installation and first-time setup

This guide is for **operators** deploying Ping Monitor for the first time on **Windows Server + IIS** from a release package.

It covers the complete first-run flow:

- web app installation in IIS
- startup gate completion (database + schema + initial admin)
- first agent package deployment
- notification infrastructure and per-user notification setup

---

## 1) Deployment overview

Ping Monitor deploys as:

- **Control plane web app**: ASP.NET Core app hosted behind IIS.
- **Execution plane agent packages**: generated by the web app and downloaded from **Agents > Deploy new agent**.
- **Database backend**: **MySQL only**.
- **Optional notification integrations**: SMTP and Telegram infrastructure (admin) plus per-user preferences.

The platform is observability/alerting only: agents submit raw check results; server-side logic calculates state and alerts.

---

## 2) Prerequisites

## Windows and hosting prerequisites

- Windows Server with IIS support (production deployment target).
- IIS installed.
- ASP.NET Core hosting support installed on the server:
  - If using framework-dependent publish, install the **.NET 10 Hosting Bundle**.
  - If deploying the repository release ZIP (self-contained app), runtime is bundled, but IIS ANCM hosting support is still required.

> If Hosting Bundle was installed before IIS, reinstall/repair Hosting Bundle after IIS is enabled.

## Database prerequisites

- Reachable **MySQL 8.0** server.
- Database/network path open from IIS host to MySQL (default port 3306 unless changed).
- Credentials with permissions to create/update schema objects for first startup.

## Network and certificate prerequisites

- Public/operator access to web app over **HTTPS**.
- Agent connectivity model is outbound-only from agent to server.
- `Agent provisioning site URL` must be an HTTPS URL that agent hosts can actually reach.

## Inbound port expectations

- Web app inbound: HTTPS TCP 443 (recommended/expected).
- No inbound ports are required on agent hosts.

---

## 3) Install IIS and required Windows features

On the Windows host:

1. Install **Web Server (IIS)** role.
2. Enable common IIS role services needed for ASP.NET Core hosting:
   - Web Server
   - Static Content
   - Default Document
   - HTTP Errors
   - Request Filtering
3. Install/repair **ASP.NET Core Hosting Bundle (.NET 10)** to register ANCM (`AspNetCoreModuleV2`).
4. Restart IIS (`iisreset`) after Hosting Bundle installation.

Verification check:

- Open IIS Manager and confirm site creation is available.
- Confirm server has ANCM support (required by `web.config` in publish output).

---

## 4) Prepare the release package

From the official release ZIP (`PingMonitor-<Version>-win-x64.zip`), extract to a staging folder.

Expected top-level layout:

- `app/` → deployable IIS web app content (this is what IIS serves)
- `docs/` → operational documentation shipped with release
- `config-samples/` → sample configuration references
- `manifest.json` → release metadata

## What to copy to IIS

Copy **contents of `app/`** to your IIS site physical path (for example `C:\inetpub\PingMonitor\site`).

## What not to edit before first run

- Do not hardcode DB credentials into code.
- Do not bypass startup gate.
- Do not remove `web.config` from `app/`.

## What you may adjust after deployment

- `appsettings.json` operational values (if needed), while preserving startup gate flow.
- IIS bindings, app pool identity, and filesystem ACLs.

## 5) Database setup (actual runtime behavior)

Ping Monitor startup requires valid MySQL config and schema.

On first run, startup gate checks:

1. database configuration present
2. DB connectivity success
3. schema compatibility with required version
4. at least one `Admin` user exists

If any check fails, app stays in startup gate mode and redirects to `/startup-gate`.

## How database config is stored

- Stored by startup gate in `App_Data/StartupGate` (under site content root by default).
- Settings file: `dbsettings.json` (host/port/db/user).
- Password file: `dbpassword.bin`.
- On Windows, password is protected with DPAPI machine scope.
- The startup gate UI never displays saved password back.

## Connection details you must provide in startup gate

- Host
- Port (default 3306)
- Database name
- Username
- Password

## Schema behavior

- Schema creation/upgrades are **explicit** (button-driven in startup gate).
- Schema changes are **not auto-applied** during normal startup.

## 6) IIS deployment procedure

1. Extract release ZIP.
2. Create deploy folder, for example:
   - `C:\inetpub\PingMonitor\site`
3. Copy `app\*` into that folder.
4. In IIS Manager, create a dedicated application pool:
   - .NET CLR: **No Managed Code**
   - Pipeline: Integrated
   - Identity: dedicated service identity recommended
5. Create IIS site pointing to the physical path.
6. Configure HTTPS binding (hostname + certificate).
7. Start site.

## File permissions required

The app pool identity must have read/write access to:

- site root (read)
- `App_Data/StartupGate` (create/read/write)
- any other configured `App_Data/*` paths used by runtime features (for example backups)

If write permission is missing, startup gate save/apply actions will fail.

## Forwarded header note (important behind reverse proxies)

If IIS is behind another reverse proxy/load balancer, configure trusted forwarded headers (`ForwardedHeaders:KnownProxies` / `KnownNetworks`) so scheme/host and local-vs-remote checks remain correct.

---

## 7) First launch and startup gate completion

After IIS starts the site:

1. Open site locally on the IIS server using loopback (`https://localhost`, `http://127.0.0.1`, etc.).
2. Go to `/startup-gate`.
3. Complete in order:
   - **Save MySQL configuration**
   - **Create / apply required schema**
   - **Create initial admin**
4. Confirm diagnostics all pass and app exits startup gate.
5. Confirm `/` or `/status` loads normally.

Important access rule:

- startup gate write actions are local-only.
- remote users can view diagnostics but cannot save/apply/bootstrap.

## 8) First agent deployment

## Why `Agent provisioning site URL` matters

When you generate an agent package, the web app writes this URL into agent `.env` as `SERVER_URL`.

If URL is wrong, agent cannot call back to:

- `/api/v1/agent/hello`
- `/api/v1/agent/config`
- `/api/v1/agent/heartbeat`
- `/api/v1/agent/results`

It must be:

- absolute
- HTTPS
- reachable from that agent host/network

## Steps

1. Sign in as admin.
2. Go to **Agents > Deploy new agent** (`/agents/deploy`).
3. In **Agent provisioning site URL**, set correct HTTPS base URL and save.
4. Enter **Agent name**.
5. Click **Create and download package**.
6. Extract downloaded `ping-agent-<instance>-<timestamp>.zip` on agent host.
7. Run `run-agent.cmd` (Windows helper) or run Python entry point manually.

Package includes:

- `app/*.py`
- `requirements.txt`
- `run-agent.cmd`
- generated `.env` containing `SERVER_URL`, `INSTANCE_ID`, `API_KEY`

## Verify first agent is connected

In **Agents > Manage agents** (`/agents`), confirm:

- agent record exists
- status/last seen/last heartbeat updates over time
- API key state is active

Then create endpoint + assignment and confirm check events appear in status/history.

## 9) Notification infrastructure settings (admin)

Go to **Admin > Notification infrastructure settings** (`/admin/notification-infrastructure-settings`).

## SMTP transport settings

Configure:

- Enable SMTP notifications
- SMTP host
- SMTP port
- TLS on/off
- Username (optional depending on relay)
- Password (stored secret; can be retained or cleared)
- From address (required)
- From display name (optional)

Use **Send SMTP test email to my account** to validate transport.

SMTP test requires signed-in admin to have a verified email on profile.

## Telegram transport settings

Configure:

- Enable Telegram infrastructure
- Inbound mode (Polling is current supported mode)
- Poll interval value (stored setting)
- Bot token (stored secret; can be retained or cleared)

Phase-1 production expectation: use polling mode.

---

## 10) User notification settings (per operator)

Each user configures their own preferences at:

- **Profile > Notification settings** (`/profile/notification-settings`)

Per-user controls include:

- Browser notifications + event toggles
- SMTP notifications + event toggles
- Telegram notifications + event toggles
- Quiet hours and per-channel quiet-hour suppression toggles

## Telegram linking flow (per user)

1. In profile notification settings, click **Generate link code**.
2. Send the 8-digit code to configured bot in private chat.
3. Wait for polling worker to process message.
4. Confirm account is shown as linked.

## How global and user settings interact

Delivery requires both:

- infrastructure channel enabled/configured by admin (SMTP/Telegram), and
- per-user channel + event toggles enabled.

For SMTP, user also needs verified email.

## 11) Deployment verification checklist

Use this post-install checklist:

- [ ] Site loads via HTTPS.
- [ ] Admin can sign in.
- [ ] Startup gate no longer active after bootstrap.
- [ ] DB connectivity and schema are healthy.
- [ ] Agent provisioning URL saved and valid HTTPS.
- [ ] First agent package downloads successfully.
- [ ] First agent shows heartbeat/last seen in `/agents`.
- [ ] Endpoint assignment produces status/events.
- [ ] SMTP test send succeeds (if SMTP enabled).
- [ ] Telegram bot token saved and user linking succeeds (if Telegram enabled).
- [ ] User receives at least one eligible notification event (or test equivalent).

## 12) Troubleshooting (first-time install)

## App always shows startup gate

One or more required checks still failing. Review diagnostics on `/startup-gate` and complete missing step (DB config, connectivity, schema apply, admin bootstrap).

## Cannot save startup gate actions remotely

Expected behavior. Use local loopback access on the IIS server for write operations.

## DB connection/config errors

Re-check host, port, db name, username, password; verify firewall path from IIS host to MySQL server.

## Agent package generation blocked

Check `Agent provisioning site URL` on `/agents/deploy`:

- must be present
- must be absolute
- must use HTTPS

## Agent fails to connect back

Validate generated `.env` values (`SERVER_URL`, `INSTANCE_ID`, `API_KEY`) and verify agent host can reach server URL over HTTPS.

## Notifications not sending

- SMTP: verify infrastructure settings + verified user email + per-user SMTP toggles.
- Telegram: verify bot token, polling enabled, user linked via code, and per-user Telegram toggles.
- Browser: app must be open and browser permission granted.

## IIS permission/path issues

Ensure app pool identity can write to `App_Data/StartupGate` and other configured writable directories.
