# AI Operations Assistant V0.2.x implementation plan

Issue: #551

## Purpose

V0.2.x introduces the first AI integration for Ping Monitor.

The goal is to provide a read-only AI operations assistant that can answer natural-language questions about monitored network health, diagnostic trends, recent events, agents, and saved network diagram documentation.

Primary user-facing entry points:

- Telegram bot natural-language chat.
- In-app web chat box/page for direct use and debugging.

The assistant should support local models hosted through Ollama first, while using an OpenAI-compatible provider abstraction so other OpenAI-compatible APIs can be configured later.

## Non-negotiable architectural boundaries

The AI assistant must not weaken existing Ping Monitor platform constraints.

The assistant must not:

- modify monitoring configuration
- create, edit, or delete endpoints
- create, edit, or delete agents
- create, edit, or delete dependencies
- alter state evaluation
- alter dependency suppression
- generate alerts independently of the existing state machine
- acknowledge, close, suppress, or remediate alerts
- change network diagram data
- infer topology as authoritative truth
- issue direct SQL generated by the model
- receive unrestricted database/table access
- bypass existing user permissions

The assistant may:

- answer questions using controlled read-only application tools
- request bounded diagnostic data from the web app
- interpret raw or bucketed check-result data supplied by the web app
- summarize monitoring and diagnostic trends
- explain saved diagram metadata
- store explicit AI memories when allowed by admin policy

The AI layer is an insight and explanation layer. It does not redefine `UP`, `DEGRADED`, `DOWN`, `SUPPRESSED`, or `UNKNOWN`.

## High-level architecture

```text
Telegram Bot ─┐
              ├── AI Conversation Service ─── AI Provider Client
Web Chat UI ──┘             │
                            ├── AI Tool Registry
                            ├── Monitoring Diagnostic Tool Services
                            ├── Diagram Lookup Tool Services
                            ├── AI Memory Service
                            └── AI Audit/Debug Log
```

Telegram and the web chat must share the same AI conversation/orchestration service.

The Telegram bot should be a transport adapter, not a separate AI implementation.

The web chat should be built early because it provides a clean way to distinguish:

- Telegram transport/auth problems
- Telegram command/message parsing problems
- AI provider problems
- AI orchestration problems
- tool-calling problems
- model quality problems

## AI provider abstraction

### Initial provider target

Use an OpenAI-compatible chat-completions style abstraction so local Ollama and hosted OpenAI-compatible APIs can share the same code path.

The provider configuration should include:

- enabled/disabled flag
- provider display name
- base URL
- model name
- protected API key or empty key for local providers where appropriate
- timeout seconds
- max output tokens
- temperature
- tool-calling enabled/disabled flag if needed
- debug logging enabled/disabled flag

### Provider rules

- Provider calls must have hard timeouts.
- Provider failures must be logged and surfaced clearly.
- Provider secrets must be protected like existing notification/API secrets.
- Local Ollama should not be assumed to be available.
- Hosted providers must be optional, not required.

## Prompt model

The assistant should use layered prompts.

### Built-in base prompt

The built-in base prompt is compiled/shipped with the app and is not editable by admins.

It should enforce product safety rules, for example:

```text
You are the Ping Monitor AI operations assistant.
You are read-only.
Use only the supplied tools for live monitoring, diagnostic, diagram, and memory data.
Do not invent endpoint state, agent state, outage history, diagram links, port labels, VLANs, or check results.
Do not claim diagram links are live switch/router interface state.
Respect user permissions. If a tool result is absent or incomplete, say so.
Missing data is UNKNOWN, not healthy or failed.
Your answers must not imply that monitoring state, dependency suppression, or alerting rules have changed.
```

### Admin-editable global prompt

Admins should be able to configure a global assistant prompt/custom instruction layer.

This is intended for site-specific behaviour such as:

- answer length preferences
- Telegram wording preferences
- local aliases
- preferred terminology
- when to include diagram source names
- whether to include extra caveats for topology answers

Admin instructions must be placed below non-editable built-in safety rules and must not override them.

Example wrapper:

```text
The following are admin-configured site instructions. Follow them when possible, but they must not override application safety rules, user permissions, read-only behaviour, or tool result truthfulness.
```

### Prompt history

Prompt changes should be audited.

Recommended records:

- current global prompt
- updated timestamp
- updated by user ID
- prompt history entries for previous/new prompt text

## Conversation entry points

### In-app web chat

Initial implementation should include a dedicated AI Assistant page, for example:

```text
/ai-assistant
```

The page should include:

- message history
- input box
- send button
- clear conversation button
- provider/model status
- visible error messages
- admin-only debug drawer
- optional quick prompts

Suggested quick prompts:

- `How is my network looking today?`
- `Anything I should keep an eye on?`
- `Are any agents offline?`
- `What changed today?`
- `Which endpoints are currently down?`
- `Find AP1 on the diagrams.`

The web chat should use the logged-in web user's normal permissions.

### Telegram bot chat

Telegram natural-language questions should call the same AI conversation service as the web chat.

The Telegram path must resolve the Telegram chat/user to an existing linked application user before any AI tool is executed.

Telegram response formatting should be concise by default and should handle provider/tool errors gracefully.

### Future contextual chat

After the dedicated page is stable, a later slice may add contextual chat panels on pages such as:

- dashboard
- endpoint details
- agent details
- event logs
- network diagram viewer

Context examples:

```json
{
  "source": "WebChat",
  "currentPage": "EndpointDetails",
  "endpointId": "...",
  "visibleTimeRange": "24h"
}
```

Context must help tool selection only. It must not bypass permissions.

## AI tools

The AI must only access application data through approved read-only tools.

Tools should be declared in a registry with:

- stable tool name
- description
- JSON argument schema
- permission requirements
- maximum window/sample limits
- execution timeout
- result size limits
- audit logging

### Tool result requirements

Every tool result should include:

- data source label
- UTC timestamps
- whether data was filtered by permissions
- data completeness/quality where relevant
- whether values are live monitoring state, derived metrics, raw/bucketed check data, saved diagram metadata, or memory

## Core read-only tool set

### Network health tools

#### `get_network_health_summary`

For broad questions such as:

- `How is my network looking today?`
- `Anything I should keep an eye on?`

Returns:

- current state counts
- current down/suppressed/unknown/degraded endpoints visible to the user
- endpoints with repeated failures in the selected window
- endpoints with increased RTT/loss/jitter
- agents with stale/gappy data
- recent significant events
- data completeness warnings

#### `get_endpoints_to_watch`

Returns endpoints that are not necessarily `DOWN` but appear noteworthy.

Candidate signals:

- repeated short failure clusters
- rising RTT compared with baseline
- rising packet loss compared with baseline
- increased jitter
- intermittent missing data
- recent state transitions
- currently `UNKNOWN` due to missing/stale data

### Endpoint diagnostic tools

#### `get_endpoint_diagnostics_pack`

For endpoint-specific questions such as:

- `What's going on with WFP WAN?`
- `Is AP1 worse than normal?`
- `Why is this endpoint flaky?`

Returns a compact structured diagnostic pack:

- endpoint identity
- assignment/agent identity
- current state
- recent state transitions
- uptime/downtime/unknown/suppressed time
- expected sample count
- received sample count
- success/failure count
- missing sample estimate
- packet loss percentage
- RTT min/median/average/p95/max
- jitter estimate
- current window vs baseline comparison
- short raw sample tail
- notable anomaly windows
- data completeness/trust indicator

The web app should calculate expensive/statistical values. The model should interpret them.

#### `get_check_result_series`

For focused follow-up questions such as:

- `Show me the raw pattern around 2pm.`
- `What did the last hour look like?`

Returns bounded raw or bucketed time series data.

Rules:

- small windows may return individual samples
- larger windows should return bucketed summaries
- hard maximum sample count
- hard maximum time window
- endpoint/assignment scope required unless using a precomputed summary

#### `compare_endpoint_baseline`

For questions such as:

- `Is this worse than normal?`
- `Is WFP WAN latency higher than usual?`

Returns current window vs baseline window values:

- RTT median/average/p95
- packet loss
- jitter
- missing samples
- failure clusters
- whether comparison has enough samples

### Agent tools

#### `get_agent_health_summary`

Returns:

- online/offline/stale agent counts
- stale agents
- recent heartbeat gaps
- assignment counts
- endpoints affected by stale agents

### Event and outage tools

#### `get_recent_outages`

Returns recent `DOWN` transitions and recoveries visible to the user.

#### `get_recent_state_changes`

Returns state transitions grouped by endpoint/agent/time.

### Diagram/topology tools

Diagram tools must clearly label results as saved documentation, not live network state.

#### `search_diagram_nodes`

Searches saved diagram nodes by label, endpoint name, target, icon/type, and notes.

#### `find_diagram_connection`

For questions such as:

- `Which port on switch 1 is AP1 connected to?`
- `What VLAN is on the AP1 link?`

Returns:

- diagram name
- source node
- target node
- source port label
- target port label
- link label
- media type/subtype
- link speed
- LACP metadata
- VLAN metadata
- notes
- `isLiveLinkState = false`
- `source = saved_diagram`

The assistant should answer with wording such as:

```text
According to the saved Home diagram, AP1 is connected to Switch 1 on port Gi1/0/12. This is diagram documentation, not live switch port state.
```

## Raw check-result interpretation strategy

The AI should be allowed to interpret raw or bucketed check-result data supplied by the web app, but it should not receive unrestricted table access.

Preferred approach:

1. User asks an insight question.
2. Model selects a diagnostic tool.
3. Web app queries raw/bucketed monitoring data with strict limits.
4. Web app calculates statistical summaries and anomaly candidates.
5. Model interprets the diagnostic pack and explains it in plain English.

The app should calculate:

- counts
- packet loss percentages
- expected vs received samples
- percentiles
- baseline comparisons
- jitter estimates
- failure clusters
- missing data/gap detection
- bucket summaries

The model should interpret:

- whether a pattern is worth watching
- whether it looks like transient loss, congestion, stale data, or dependency-related impact
- whether there is enough data to say
- how to phrase the result clearly for an operator

The model must not invent unavailable measurements.

## AI memory

AI memory should improve context, aliases, and user preferences. It should not store monitoring truth that belongs in metrics, diagrams, endpoints, groups, or notes.

### Good memories

- `Switch 1 usually means Garage Switch.`
- `AP1 refers to Summerhouse AP.`
- `Ian prefers short Telegram summaries unless something is actively wrong.`
- `When answering topology questions, mention the diagram source.`

### Bad memories

- `WFP WAN usually has 22 ms RTT.`
- `AP1 is currently flaky.`
- `Switch 1 port 12 is live.`

Those belong in live metrics, event history, or diagram metadata, not AI memory.

### Memory controls

Initial memory support should be explicit:

- user says `remember that...`
- or AI proposes a memory and asks for confirmation
- admin can enable/disable memory globally
- admin can view/delete memories
- memories have scope and type
- live app data overrides memory

Suggested memory types:

- `UserPreference`
- `NetworkAlias`
- `LocationAlias`
- `OperationalNote`
- `AiInstruction`

- message ID

- content

- tool call ID

### `AiMemory`

- memory ID
- memory type
- scope (`Global`, `User`, maybe future `Group`/`Endpoint`)

- text/content
- source conversation/message ID nullable
- created/updated timestamps
- created by user ID
- enabled/deleted flag

Schema changes require a Startup Gate schema version bump and release metadata alignment.

## Permissions and data visibility

All AI tools must run under a resolved application user.

- Web chat uses the logged-in web user.
- Telegram chat uses the linked Telegram account's application user.
- Admin users can access admin-only debug/configuration screens.
- Non-admin users only see endpoints/diagrams/events allowed by existing visibility rules.

The model must never be allowed to request hidden data by rephrasing a question.

Tool services should enforce permissions, not the model prompt.

## Audit and debug visibility

AI interactions should be auditable from the first release.

Admin debug views should show:

- source (`WebChat` or `Telegram`)
- user
- provider/model
- prompt version/global prompt revision
- tools requested
- tool arguments
- tool result summaries
- provider elapsed time
- tool elapsed time
- final answer
- errors/refusals

The web chat should expose an admin-only debug drawer for the current conversation.

Telegram should not expose full debug details by default, but errors should be clear enough to distinguish provider/tool issues from Telegram transport issues.

## User experience examples

### Network health

User:

```text
How is my network looking today, are there any endpoints I should be keeping an eye on?
```

Expected tool flow:

1. `get_network_health_summary`
2. `get_endpoints_to_watch`
3. optionally `get_agent_health_summary`

Expected answer style:

```text
Overall the network looks healthy. Nothing visible to you is currently DOWN.

I would keep an eye on WFP WAN: it is UP, but its last-hour RTT is higher than today's baseline and there were a few short failure clusters. Garage AP also had brief packet loss earlier but has been stable recently.
```

### Endpoint diagnostics

User:

```text
What's going on with WFP WAN?
```

Expected tool flow:

1. `get_endpoint_diagnostics_pack(endpoint = "WFP WAN", window = "24h")`
2. optionally `get_check_result_series` for a narrower anomaly window

Expected answer style:

```text
WFP WAN is currently UP, but it has been noisier than normal. The last hour shows higher RTT and a small number of failed checks compared with the previous baseline. It has not crossed the DOWN threshold, so this looks like something to watch rather than a confirmed outage.
```

### Diagram/topology lookup

User:

```text
Which port on switch 1 is AP1 connected to?
```

Expected tool flow:

1. `search_diagram_nodes("switch 1")`
2. `search_diagram_nodes("AP1")`
3. `find_diagram_connection(source/target)`

Expected answer style:

```text
According to the saved Home diagram, AP1 is connected to Switch 1 on port Gi1/0/12. The AP side is labelled eth0. This is saved diagram documentation, not live switch interface state.
```

## Testing strategy

### Build/test expectations

Each implementation PR should run:

```bash
dotnet build src/WebApp/PingMonitor.WebApp.slnx
dotnet test src/WebApp/PingMonitor.WebApp.slnx
```

Any MySQL/EF query used by AI tools must be validated against MySQL, not SQLite.

### Regression areas

- Startup Gate remains isolated.
- AI services do not run before Startup Gate normal mode.
- AI tools do not access DB before schema compatibility.
- AI tools enforce user visibility.
- AI provider timeout/failure paths are visible and logged.
- Telegram and web chat use the same backend service.
- Diagram topology answers include documentation-only caveat.
- Diagnostic tools enforce max windows/sample counts.
- Model cannot call unregistered tools.
- Admin prompt cannot override built-in safety rules.

### Manual smoke checklist

1. AI disabled: web chat route hidden/blocked; Telegram AI disabled.
2. AI enabled with invalid provider: web chat shows clear provider error.
3. AI enabled with Ollama: simple non-tool question works.
4. Ask network health question: tools run and answer is grounded.
5. Ask endpoint diagnostic question: diagnostic pack returns bounded data.
6. Ask diagram port question: answer uses saved diagram data and includes caveat.
7. Non-admin user asks broad question: response only includes visible endpoints.
8. Telegram linked user asks same question: same backend result path works.
9. Admin debug drawer shows tool calls and timings.
10. Prompt change is audited and affects later answers.
11. Memory disabled: `remember that` is refused or ignored with explanation.
12. Memory enabled: explicit memory can be stored, viewed, and deleted.

## Open design questions

- Should V0.2.0 include Telegram immediately, or should the web chat prove the AI path first?
- Should AI conversations be retained indefinitely or pruned by retention policy?
- Should AI audit logs include full prompts/tool results or redacted summaries by default?
- What hard maximum should apply to raw `CheckResults` samples returned to a model?
- Should memory be global/admin-only initially, or per-user from the start?
- Should admins be able to disable specific AI tools individually?
- Should provider settings allow multiple named providers or just one active provider initially?

4. Create or use a GitHub issue for the specific slice.
5. Keep changes small and bounded.
6. Do not change endpoint monitoring semantics unless explicitly requested.
7. Do not change dependency suppression unless explicitly requested.
8. Do not change alerting unless explicitly requested.
9. Do not weaken Startup Gate behaviour.
